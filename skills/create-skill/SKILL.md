# Skill 创建器

用于创建新 skill 并持续迭代改进的工具。

整体流程如下：

- 确定 skill 要做什么，以及大致如何实现
- 写出初稿
- 准备几个测试提示词，在配置了该 skill 的 AI 代理上运行
- 与用户一起定性和定量评估结果
  - 在运行进行中，起草一些定量评估指标（如果已有，可直接使用或按需修改），然后向用户解释（或解释已有的指标）
  - 使用 `eval-viewer/generate_review.py` 脚本向用户展示结果，同时展示定量指标
- 根据用户对结果的反馈重写 skill（以及定量基准测试中发现的明显问题）
- 反复迭代直到满意
- 扩大测试集，以更大规模再次测试

你的任务是判断用户处于哪个阶段，然后介入并帮助推进。例如，用户说"我想做一个 X 的 skill"，你可以帮助明确需求、写初稿、编写测试用例、确定评估方式、运行所有提示词，然后迭代。

也可能用户已经有了草稿，这时直接进入评估/迭代环节即可。

当然，要保持灵活——如果用户说"不用跑那么多评估，跟着感觉走就好"，那就这样做。

skill 完成后（顺序可灵活调整），还可以运行 skill 描述优化器，我们有专门的脚本，用来优化 skill 的触发效果。

好，开始吧。

## 与用户沟通

skill 创建器的用户对编程术语的熟悉程度可能差异很大。现在有一种趋势——AI 代理的能力正在激励水管工打开终端、父母祖父母 google "如何安装 npm"。当然，大多数用户还是有一定计算机基础的。

请注意上下文线索，理解如何措辞！默认情况下参考：

- "evaluation"（评估）和 "benchmark"（基准测试）基本可以直接用
- "JSON" 和 "assertion"（断言）这类词，要先看到用户明确知道这些概念的信号，否则需要解释

如有疑问，简短解释一下术语是没问题的。

## 创建 skill

### 捕捉意图

首先理解用户的意图。当前对话中可能已经包含用户想要固化的工作流（例如用户说"把这个变成一个 skill"）。如果是这样，先从对话历史中提取——使用了哪些工具、步骤顺序如何、用户做了哪些修正、观察到的输入/输出格式。用户可能需要补充缺失信息，并在进入下一步前确认。

1. 这个 skill 应该让 AI 代理做什么？
2. 何时应该触发这个 skill？（哪些用户用语/场景）
3. 期望的输出格式是什么？
4. 是否需要设置测试用例来验证 skill 的效果？输出可客观验证的 skill（文件转换、数据提取、代码生成、固定流程步骤）适合设置测试用例；输出主观的 skill（写作风格、艺术创作）通常不需要。根据 skill 类型给出建议默认值，但由用户决定。

### 访谈与调研

主动询问边界情况、输入/输出格式、示例文件、成功标准和依赖项。在这部分理清楚之前，不要急着写测试提示词。

检查可用的 MCP——如果对调研有帮助（搜索文档、查找类似 skill、查阅最佳实践），在条件允许时通过子 agent 并行调研，否则内联进行。提前准备好上下文，减少对用户的负担。

### 编写 SKILL.md

根据用户访谈，填写以下内容：

- **name**：skill 标识符
- **description**：何时触发、做什么。这是主要触发机制——既要包含 skill 的功能，也要包含具体的使用场景。所有"何时使用"的信息放在这里，不放在正文中。注意：AI 代理目前倾向于"触发不足"——在本该使用 skill 时不去使用。为了应对这一点，请让 skill 描述稍微"积极主动"一些。例如，不要写"如何构建简单快速的仪表板来展示内部数据"，而是写"如何构建简单快速的仪表板来展示内部数据。只要用户提到仪表板、数据可视化、内部指标或想展示任何公司数据，即使没有明确要求'仪表板'，也务必使用此 skill。"
- **compatibility**：所需工具、依赖项（可选，很少需要）
- **其余 skill 内容 :)**

### Skill 编写指南

#### Skill 的结构

```
skill-name/
├── SKILL.md（必须）
│   ├── YAML frontmatter（name、description 必填）
│   └── Markdown 说明
└── 捆绑资源（可选）
    ├── scripts/    - 用于确定性/重复性任务的可执行代码
    ├── references/ - 按需加载到上下文的文档
    └── assets/     - 输出中使用的文件（模板、图标、字体）
```

#### 渐进式披露

Skill 使用三级加载系统：

1. **元数据**（name + description）——始终在上下文中（约 100 词）
2. **SKILL.md 正文**——skill 触发时在上下文中（理想情况下 <500 行）
3. **捆绑资源**——按需加载（无限制，脚本可在不加载的情况下执行）

这些字数是大致参考，如有需要可以适当超出。

**关键模式：**

- 保持 SKILL.md 在 500 行以内；如果接近上限，增加一层层级并清晰指引模型应在何处继续查阅
- 在 SKILL.md 中清晰引用文件，并说明何时应该读取它们
- 对于大型参考文件（>300 行），包含目录

**领域组织**：当 skill 支持多个领域/框架时，按变体组织：

```
cloud-deploy/
├── SKILL.md（工作流 + 选择逻辑）
└── references/
    ├── aws.md
    ├── gcp.md
    └── azure.md
```

AI 代理只读取相关的参考文件。

#### 不出意料原则

不言而喻，skill 不得包含恶意软件、漏洞利用代码或任何可能危害系统安全的内容。skill 的内容在描述时不应让用户感到意外。不要接受创建误导性 skill 或旨在促进未授权访问、数据泄露或其他恶意活动的 skill 的请求。不过，类似"扮演 XYZ"的角色扮演是可以的。

#### 编写模式

说明中优先使用祈使句形式。

**定义输出格式**——可以这样做：

```markdown
## 报告结构

始终使用以下精确模板：

# [标题]

## 执行摘要

## 关键发现

## 建议
```

**示例模式**——包含示例很有用。可以这样格式化（但如果示例中有"输入"和"输出"，可以稍作调整）：

```markdown
## 提交信息格式

**示例 1：**
输入：Added user authentication with JWT tokens
输出：feat(auth): implement JWT-based authentication
```

### 写作风格

尽量解释清楚为什么某件事重要，而不是使用重复强调的 MUST。运用心智理论，尽量使 skill 通用而不是过于局限于特定示例。先写初稿，然后以新鲜视角审视并改进。

### 测试用例

写完 skill 草稿后，想出 2-3 个真实的测试提示词——真正的用户会说的那种。与用户分享：[不必用这个确切措辞]"这是我想试的几个测试用例，这些合适吗？还是你想添加更多？"然后运行它们。

将测试用例保存到 `evals/evals.json`。现在先不写断言——只写提示词。下一步在运行过程中起草断言。

```json
{
  "skill_name": "example-skill",
  "evals": [
    {
      "id": 1,
      "prompt": "用户的任务提示词",
      "expected_output": "预期结果描述",
      "files": []
    }
  ]
}
```

完整 schema（包括 `assertions` 字段，后续添加）见 `references/schemas.md`。

## 运行和评估测试用例

本节是一个连续的序列——不要中途停止。不要使用 `/skill-test` 或任何其他测试 skill。

将结果放在 `<skill-name>-workspace/` 目录中，与 skill 目录并列。在 workspace 中，按迭代轮次组织结果（`iteration-1/`、`iteration-2/` 等），每个迭代内每个测试用例有自己的目录（`eval-0/`、`eval-1/` 等）。不要提前创建所有目录——边进行边创建。

### 步骤 1：在同一轮次中启动所有运行（含 skill 和基线）

对于每个测试用例，在同一轮次中启动两个子 agent——一个有 skill，一个没有。这很重要：不要先启动有 skill 的运行，再回来做基线。一次性全部启动，让它们同时完成。

**有 skill 的运行：**

```
执行以下任务：
- Skill 路径：<path-to-skill>
- 任务：<eval 提示词>
- 输入文件：<eval 文件，如果有；否则填"无">
- 将输出保存到：<workspace>/iteration-<N>/eval-<ID>/with_skill/outputs/
- 需要保存的输出：<用户关心的内容——例如".docx 文件"、"最终 CSV">
```

**基线运行**（相同提示词，但基线取决于情境）：

- **创建新 skill**：完全没有 skill。相同提示词，无 skill 路径，保存到 `without_skill/outputs/`。
- **改进现有 skill**：旧版本。编辑前先快照 skill（`cp -r <skill-path> <workspace>/skill-snapshot/`），然后让基线子 agent 指向快照。保存到 `old_skill/outputs/`。

为每个测试用例写一个 `eval_metadata.json`（断言暂时可以为空）。根据测试内容给每个 eval 取一个描述性名称——不要只用"eval-0"。目录名也使用这个名称。如果本次迭代使用了新的或修改过的 eval 提示词，为每个新 eval 目录创建这些文件——不要假设它们会从上一次迭代延续。

```json
{
  "eval_id": 0,
  "eval_name": "描述性名称",
  "prompt": "用户的任务提示词",
  "assertions": []
}
```

### 步骤 2：运行进行中，起草断言

不要等运行结束——可以利用这段时间。为每个测试用例起草定量断言，并向用户解释。如果 `evals/evals.json` 中已有断言，审查并解释它们检查的内容。

好的断言应是客观可验证的，且有描述性名称——在基准测试查看器中应能清晰读懂，让人一眼就明白每条断言在检查什么。主观 skill（写作风格、设计质量）更适合定性评估——不要强行给主观内容添加断言。

起草完成后，更新 `eval_metadata.json` 文件和 `evals/evals.json` 中的断言。同时向用户解释他们将在查看器中看到什么——定性输出和定量基准测试。

### 步骤 3：运行完成时记录时间数据

每个子 agent 任务完成时，你会收到包含 `total_tokens` 和 `duration_ms` 的通知。立即将该数据保存到运行目录中的 `timing.json`：

```json
{
  "total_tokens": 84852,
  "duration_ms": 23332,
  "total_duration_seconds": 23.3
}
```

这是捕获此数据的唯一机会——它通过任务通知传来，不会持久化到其他地方。收到每个通知时立即处理，而不是尝试批量处理。

### 步骤 4：评分、汇总并启动查看器

所有运行完成后：

1. **对每次运行评分**——启动一个评分子 agent（或内联评分），读取 `agents/grader.md` 并对照输出评估每个断言。将结果保存到每个运行目录中的 `grading.json`。grading.json 的 expectations 数组必须使用字段 `text`、`passed` 和 `evidence`（不是 `name`/`met`/`details` 或其他变体）——查看器依赖这些确切的字段名。对于可以编程检查的断言，编写并运行脚本而不是目测——脚本更快、更可靠，且可在迭代间复用。

2. **汇总为基准测试**——从 skill-creator 目录运行汇总脚本：

   ```bash
   python -m scripts.aggregate_benchmark <workspace>/iteration-N --skill-name <name>
   ```

   这会生成 `benchmark.json` 和 `benchmark.md`，包含每个配置的 pass_rate、时间和 token，带均值 ± 标准差和差值。如果手动生成 benchmark.json，查看器期望的确切 schema 见 `references/schemas.md`。
   将每个 with_skill 版本放在其基线版本之前。

3. **进行分析**——读取基准测试数据，找出汇总统计可能掩盖的规律。参见 `agents/analyzer.md`（"分析基准测试结果"部分）了解要查找的内容——例如无论是否有 skill 都能通过的断言（无区分度的）、高方差 eval（可能不稳定），以及时间/token 权衡。

4. **启动查看器**，同时展示定性输出和定量数据：

   ```bash
   nohup python <skill-creator-path>/eval-viewer/generate_review.py \
     <workspace>/iteration-N \
     --skill-name "my-skill" \
     --benchmark <workspace>/iteration-N/benchmark.json \
     > /dev/null 2>&1 &
   VIEWER_PID=$!
   ```

   第 2 次及以后的迭代，还要传入 `--previous-workspace <workspace>/iteration-<N-1>`。

   **Cowork / 无界面环境：** 如果 `webbrowser.open()` 不可用或环境没有显示器，使用 `--static <output_path>` 写入独立 HTML 文件，而不是启动服务器。用户点击"提交所有评审"时会下载 `feedback.json` 文件。下载后，将 `feedback.json` 复制到 workspace 目录，供下一次迭代使用。

   注意：请使用 generate_review.py 创建查看器；无需编写自定义 HTML。

5. **告知用户**，类似："我已在浏览器中打开了结果。有两个标签页——'输出'让你逐个点击测试用例并留下反馈，'基准测试'显示定量对比。完成后，回来告诉我。"

### 用户在查看器中看到的内容

"输出"标签页每次显示一个测试用例：

- **提示词**：给出的任务
- **输出**：skill 产生的文件，尽可能内联渲染
- **上一次输出**（第 2+ 次迭代）：折叠显示上次迭代的输出
- **正式评分**（如果运行了评分）：折叠显示断言通过/失败情况
- **反馈**：自动保存的文本框
- **上一次反馈**（第 2+ 次迭代）：上次的评论，显示在文本框下方

"基准测试"标签页显示统计摘要：每个配置的通过率、时间和 token 使用情况，包含 per-eval 细分和分析观察。

通过上一个/下一个按钮或方向键导航。完成后，点击"提交所有评审"，所有反馈保存到 `feedback.json`。

### 步骤 5：读取反馈

当用户告诉你完成后，读取 `feedback.json`：

```json
{
  "reviews": [
    {
      "run_id": "eval-0-with_skill",
      "feedback": "图表缺少坐标轴标签",
      "timestamp": "..."
    },
    { "run_id": "eval-1-with_skill", "feedback": "", "timestamp": "..." },
    {
      "run_id": "eval-2-with_skill",
      "feedback": "完美，非常喜欢",
      "timestamp": "..."
    }
  ],
  "status": "complete"
}
```

空反馈意味着用户认为没问题。重点改进用户有具体意见的测试用例。

完成后关闭查看器服务器：

```bash
kill $VIEWER_PID 2>/dev/null
```

## 改进 skill

这是循环的核心。你已经运行了测试用例，用户已经审查了结果，现在需要根据反馈让 skill 更好。

### 如何思考改进

1. **从反馈中总结规律。** 这里的大局观是：我们试图创建能被使用数百万次的 skill（也许真的会，甚至更多），跨越许多不同的提示词。你和用户在这里反复迭代几个示例是因为这样更快。用户对这些示例了如指掌，可以快速评估新输出。但如果你们共同开发的 skill 只对这些示例有效，那就没用了。与其做些零碎的过拟合改动，或施加压迫性的约束，不如尝试不同的比喻，或推荐不同的工作模式。尝试代价相对低廉，也许会有意外收获。

2. **保持提示词精简。** 删除没有发挥作用的内容。确保阅读执行记录，而不仅仅是最终输出——如果看起来 skill 让模型浪费了大量时间做无用功，可以尝试删除 skill 中导致这种情况的部分，看看会发生什么。

3. **解释原因。** 尽力解释你要求模型做某件事的**原因**。当代 LLM 很聪明。它们有良好的心智理论，在给定良好框架时，能够超越死记硬背的指令，真正让事情发生。即使用户的反馈简短或沮丧，也要真正理解任务和用户的意图，然后将这种理解传递到说明中。如果你发现自己在大写字母中写 ALWAYS 或 NEVER，或使用超级严格的结构，这是一个黄色警告——如果可能，重新框架并解释原因，让模型理解为什么所要求的事情很重要。这是一种更人性化、更强大、更有效的方法。

4. **寻找测试用例中的重复工作。** 阅读测试运行的执行记录，注意子 agent 是否都独立编写了类似的辅助脚本或采取了相同的多步骤方法。如果所有 3 个测试用例都导致子 agent 编写了 `create_docx.py` 或 `build_chart.py`，这是一个强烈信号，表明 skill 应该捆绑该脚本。写一次，放在 `scripts/` 中，并告诉 skill 使用它。这样每次未来的调用都不必重新发明轮子。

这个任务相当重要，你的思考时间不是瓶颈；慢慢来，真正深入思考。建议先写一个修订草稿，然后重新审视并改进。真正进入用户的思维，理解他们想要和需要什么。

### 迭代循环

改进 skill 后：

1. 将改进应用到 skill
2. 将所有测试用例重新运行到新的 `iteration-<N+1>/` 目录中，包括基线运行。如果你在创建新 skill，基线始终是 `without_skill`（无 skill）——这在迭代间保持不变。如果你在改进现有 skill，根据判断决定合适的基线：用户带来的原始版本，或上一次迭代版本。
3. 使用 `--previous-workspace` 指向上一次迭代来启动评审工具
4. 等待用户审查并告诉你已完成
5. 读取新反馈，再次改进，重复

持续进行直到：

- 用户表示满意
- 所有反馈都为空（一切看起来都好）
- 没有取得有意义的进展

## 进阶：盲测比较

对于需要更严格比较两个版本 skill 的情况（例如，用户问"新版本真的更好吗？"），有一个盲测比较系统。详情请阅读 `agents/comparator.md` 和 `agents/analyzer.md`。基本思路是：将两个输出交给一个独立 agent，不告诉它哪个是哪个，让它评判质量。然后分析获胜者为何获胜。

这是可选的，需要子 agent，大多数用户不需要它。人工审查循环通常已经足够。

## 描述优化

SKILL.md frontmatter 中的 description 字段是决定 AI 代理是否调用 skill 的主要机制。创建或改进 skill 后，提议优化描述以获得更好的触发准确率。

### 步骤 1：生成触发评估查询

创建 20 个评估查询——应触发和不应触发的混合。保存为 JSON：

```json
[
  { "query": "用户提示词", "should_trigger": true },
  { "query": "另一个提示词", "should_trigger": false }
]
```

查询必须是真实的，是 AI 代理用户实际会输入的内容。不是抽象请求，而是具体详细的请求。例如，文件路径、关于用户工作或情况的个人背景、列名和值、公司名称、URL。一点点背景故事。一些可能是小写或包含缩写、错别字或口语。使用不同长度的混合，关注边界情况而不是让它们显而易见（用户有机会确认）。

不好的：`"格式化这些数据"`、`"从 PDF 提取文本"`、`"创建图表"`

好的：`"好吧，我老板刚发给我一个 xlsx 文件（在我的下载文件夹里，叫什么'Q4 sales final FINAL v2.xlsx'），她想让我添加一列显示利润率百分比。收入在 C 列，成本我想在 D 列"`

对于**应触发**的查询（8-10 个），考虑覆盖范围。你需要同一意图的不同表达方式——一些正式，一些随意。包括用户没有明确说出 skill 名称或文件类型但明显需要它的情况。加入一些不常见的用例和与其他 skill 竞争但应该获胜的情况。

对于**不应触发**的查询（8-10 个），最有价值的是接近匹配的——共享关键词或概念但实际上需要不同东西的查询。考虑相邻领域、朴素关键词匹配会触发但实际上不应触发的模糊表达，以及触及 skill 功能但在另一个工具更合适的情境中的情况。

要避免的关键点：不要让不应触发的查询明显无关。"写一个 fibonacci 函数"作为 PDF skill 的负面测试太容易了——它没有测试任何东西。负面情况应该真正有难度。

### 步骤 2：与用户一起审查

使用 HTML 模板向用户展示评估集进行审查：

1. 从 `assets/eval_review.html` 读取模板
2. 替换占位符：
   - `__EVAL_DATA_PLACEHOLDER__` → 评估项 JSON 数组（不要加引号——它是 JS 变量赋值）
   - `__SKILL_NAME_PLACEHOLDER__` → skill 名称
   - `__SKILL_DESCRIPTION_PLACEHOLDER__` → skill 当前描述
3. 写入临时文件（例如 `/tmp/eval_review_<skill-name>.html`）并打开：`open /tmp/eval_review_<skill-name>.html`
4. 用户可以编辑查询、切换应触发状态、添加/删除条目，然后点击"导出评估集"
5. 文件下载到 `~/Downloads/eval_set.json`——如果有多个版本（例如 `eval_set (1).json`），检查下载文件夹中最新的版本

这一步很重要——糟糕的评估查询会导致糟糕的描述。

### 步骤 3：运行优化循环

告诉用户："这需要一些时间——我将在后台运行优化循环并定期检查进度。"

将评估集保存到 workspace，然后在后台运行：

```bash
python -m scripts.run_loop \
  --eval-set <path-to-trigger-eval.json> \
  --skill-path <path-to-skill> \
  --model <model-id-powering-this-session> \
  --max-iterations 5 \
  --verbose
```

使用当前会话实际使用的模型 ID，这样触发测试与用户实际体验相匹配。如果当前平台不提供 `claude` CLI，此步骤需要有兼容 CLI 工具的环境才能运行。

运行期间，定期查看输出，向用户更新当前迭代和分数情况。

这会自动处理完整的优化循环。它将评估集分成 60% 训练集和 40% 保留测试集，评估当前描述（每个查询运行 3 次以获得可靠的触发率），然后调用支持扩展思考的 AI 模型，根据失败情况提出改进建议。对训练集和测试集重新评估每个新描述，最多迭代 5 次。完成后，打开 HTML 报告在浏览器中显示每次迭代的结果，并返回包含 `best_description` 的 JSON——根据测试分数而非训练分数选择，以避免过拟合。

### Skill 触发机制工作原理

理解触发机制有助于设计更好的评估查询。Skill 以 name + description 的形式出现在 AI 代理的 `available_skills` 列表中，AI 代理根据该描述决定是否查阅 skill。重要的是：AI 代理只对无法轻松独立完成的任务查阅 skill——简单的单步查询（如"读取这个 PDF"）可能不会触发 skill，即使描述完全匹配，因为代理可以用基本工具直接处理。复杂、多步骤或专业的查询在描述匹配时会可靠地触发 skill。

这意味着你的评估查询应该足够实质性，让 AI 代理确实能从查阅 skill 中受益。简单查询如"读取文件 X"是糟糕的测试用例——无论描述质量如何，它们都不会触发 skill。

### 步骤 4：应用结果

从 JSON 输出中取出 `best_description` 并更新 skill 的 SKILL.md frontmatter。向用户展示前后对比并报告分数。

## 无子 Agent 环境特定说明

在不支持子 agent 的 AI 平台（如纯 Web 对话界面）中，核心工作流是一样的（草稿 → 测试 → 审查 → 改进 → 重复），但一些机制会有所变化。以下是需要调整的地方：

**运行测试用例**：没有子 agent 意味着无法并行执行。对于每个测试用例，读取 skill 的 SKILL.md，然后按照其说明自己完成测试提示词。逐个进行。这比独立子 agent 的严格程度低（你写了 skill 同时也在运行它，所以有完整上下文），但作为健全性检查很有用——人工审查步骤可以弥补。跳过基线运行——只使用 skill 完成请求的任务。

**审查结果**：如果无法打开浏览器（例如环境没有显示器，或在远程服务器上），完全跳过浏览器审查工具。直接在对话中呈现结果。对于每个测试用例，展示提示词和输出。如果输出是用户需要查看的文件（如 .docx 或 .xlsx），将其保存到文件系统并告诉他们位置，以便下载检查。内联请求反馈："这看起来怎么样？有什么想改的吗？"

**基准测试**：跳过定量基准测试——它依赖于没有子 agent 时没有意义的基线比较。专注于用户的定性反馈。

**迭代循环**：与之前相同——改进 skill，重新运行测试用例，请求反馈——只是没有中间的浏览器审查工具。如果有文件系统，仍然可以将结果组织到迭代目录中。

**描述优化**：此部分需要 AI 平台的 CLI 工具（例如 `claude -p`），仅在支持 CLI 的环境（如 Claude Code）中可用。如果当前平台无此工具，跳过。

**盲测比较**：需要子 agent。跳过。

**打包**：`package_skill.py` 脚本在任何有 Python 和文件系统的地方都能工作。运行后用户可以下载生成的 `.skill` 文件。

## Cowork 特定说明

如果你在 Cowork，主要需要了解的是：

- 你有子 agent，所以主工作流（并行启动测试用例、运行基线、评分等）都可以工作。（但是，如果遇到严重的超时问题，可以串行运行测试提示词而不是并行。）
- 你没有浏览器或显示器，所以在生成评估查看器时，使用 `--static <output_path>` 写入独立 HTML 文件，而不是启动服务器。然后提供用户可以点击打开 HTML 的链接。
- 反馈工作方式不同：由于没有运行中的服务器，查看器的"提交所有评审"按钮会将 `feedback.json` 下载为文件。然后你可以从那里读取它（可能需要先请求访问权限）。
- 打包可以工作——`package_skill.py` 只需要 Python 和文件系统。
- 描述优化（`run_loop.py` / `run_eval.py`）在 Cowork 中应该可以正常工作，因为它通过子进程使用 `claude -p`，而不是浏览器，但请在你完全完成 skill 并且用户同意其状态良好后再使用它。
