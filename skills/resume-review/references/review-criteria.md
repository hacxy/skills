## 1. 整体结构

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 1 页精炼，信息层次清晰，模块排列合理（个人信息→技能→项目→教育），视觉重点突出 |
| 4 | 1-2 页，结构基本清晰，有少量排列可优化 |
| 3 | 2 页以内但有信息冗余，模块排列不够合理 |
| 2 | 超过 2 页，信息堆砌无主次，或关键模块缺失 |
| 1 | 结构混乱，无法快速定位核心信息 |

### 常见问题

- **篇幅过长**：超过 2 页，HR 初筛时间只有 30-60 秒，信息过多反而降低阅读效率
- **模块排列不当**：教育经历放在最前面（适用于应届生，不适用于有经验者）
- **缺少留白**：排版过于紧凑，视觉上有压迫感
- **文件名不规范**：用 `resume.pdf` 而非 `姓名_前端工程师_简历.pdf`

### 推荐模块顺序

**有经验者**：个人信息 → 专业技能 → 工作经历/项目经验 → 教育背景 → 其他（开源/博客）

**应届生**：个人信息 → 教育背景 → 专业技能 → 项目经验 → 实习经历 → 其他

## 2. 个人信息

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 包含必要信息（姓名/电话/邮箱/城市/工作年限），有 GitHub 或个人网站链接 |
| 4 | 必要信息完整，缺少作品链接 |
| 3 | 缺少 1-2 项关键信息（如城市、工作年限） |
| 2 | 关键信息缺失较多，或包含大量无关信息 |
| 1 | 信息严重不全或格式混乱 |

### 常见问题

- **缺少城市信息**：HR 筛选时需要匹配工作地点
- **缺少工作年限**：面试官需要快速判断经验层级
- **邮箱不专业**：使用 QQ 号邮箱（如 `123456@qq.com`）而非更专业的邮箱
- **照片问题**：非正式照片、过度 P 图、或不必要地占用大量版面
- **冗余信息**：政治面貌、婚姻状况、身高体重等与技术岗位无关的信息

### 前端特有加分项

- GitHub 主页链接（且有实质内容）
- 个人技术博客链接
- 在线作品 Demo 链接
- npm 包链接

## 3. 专业技能

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 技能分层清晰（精通/熟悉/了解），与工作年限匹配，每项技能都经得起追问 |
| 4 | 分层基本合理，有少量措辞需调整 |
| 3 | 未分层或分层不准确，存在轻度堆砌 |
| 2 | 严重堆砌（列出 20+ 技术词汇），或描述与年限严重不匹配 |
| 1 | 技能列表空洞无物或明显造假 |

### 常见问题

- **技能堆砌**：把所有听过的技术都列上去，面试时无法深入回答
- **分层不准确**：工作 2 年写"精通 React"，这在面试官眼中几乎等于自曝其短
- **过时技术未清理**：还写着 jQuery、AngularJS 1.x 等已经退出主流的技术
- **展开过细**：把 JS 的闭包、原型链、作用域等基础知识点逐一列出——对资深工程师来说这些是常识
- **缺少工程化技能**：只列框架不提构建工具、测试、CI/CD 等工程化能力

### 分层用词指南

| 用词 | 含义 | 面试官预期 |
|------|------|-----------|
| 精通 | 深入理解原理，能解决复杂问题 | 会追问源码实现、设计模式、性能调优细节 |
| 熟练掌握 | 大量实战经验，能独立完成复杂功能 | 会追问项目中的技术选型和权衡 |
| 熟悉 | 有实际项目经验，能独立开发 | 会问使用场景和常见 API |
| 了解 | 有基础认知，能在需要时快速上手 | 可能简单问一下概念 |

### 不同年限的技能描述参考

**1-3 年**：
> 熟练使用 Vue 3 / React 进行业务开发，了解框架核心原理（响应式系统/Fiber 架构）；熟悉 TypeScript，能在项目中规范使用类型系统；了解 Webpack/Vite 配置，有基本的构建优化经验。

**3-5 年**：
> 熟练掌握 React 技术栈（Redux/React Query/Next.js），有大型 SPA 架构设计经验；深入理解浏览器渲染机制，有 Web Vitals 优化实践（FCP/LCP/CLS）；熟悉前端工程化体系，有组件库建设和 CI/CD 流水线搭建经验。

**5 年以上**：
> 精通前端架构设计，主导过微前端/Monorepo 等大型工程方案落地；深入掌握性能优化全链路（网络层/渲染层/构建层），有可量化的优化成果；具备跨端开发经验（React Native/Flutter/Electron），能根据业务场景进行技术选型。

## 4. 项目经验

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 使用 STAR 法则，有具体量化指标，个人贡献清晰，技术选型有思考 |
| 4 | 结构基本完整，量化指标不够充分但有个人贡献描述 |
| 3 | 有项目描述但偏流水账，缺少量化和个人亮点 |
| 2 | 纯流水账，只罗列"做了什么"而不是"解决了什么问题" |
| 1 | 项目描述空洞，或明显与岗位不匹配 |

### 常见问题

- **流水账描述**："负责 XX 模块的前端开发"——这等于什么都没说
- **缺少量化指标**：没有任何数据支撑，面试官无法评估影响力
- **只有团队没有个人**："团队一起完成了 XX"——无法判断个人贡献
- **项目太多太杂**：列了 6-7 个项目，每个只有两三行——应精选 2-3 个有深度的
- **缺少技术选型理由**：只列技术栈不说为什么选——面试官会追问
- **无可访问链接**：前端项目没有 Demo 链接是严重的减分项
- **数据明显造假**：写"性能提升 90%"但无法解释具体措施

### 量化指标参考

| 维度 | 量化方式 |
|------|---------|
| 性能 | 首屏加载时间从 Xs 降到 Ys、FCP 提升 X%、LCP 从 Xs 降到 Ys |
| 体积 | 构建产物从 XMB 降到 YMB，减少 Z% |
| 效率 | 构建时间从 X 分钟降到 Y 分钟、开发效率提升 X% |
| 质量 | Bug 率降低 X%、单元测试覆盖率从 X% 提升到 Y% |
| 业务 | 用户留存率提升 X%、页面转化率提升 X%、日活从 X 增长到 Y |
| 规模 | 服务 X 万用户、支撑 X 个业务线、组件被 X 个项目复用 |
| 团队 | 带领 X 人团队、进行 X 次技术分享 |

## 5. 技术深度

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 展示了对技术原理的深入理解，有复杂问题的解决案例，有架构级思考 |
| 4 | 有一定深度的技术实践描述，但不够系统 |
| 3 | 停留在"使用"层面，未展示原理理解 |
| 2 | 技能描述全是"使用 XX 完成 YY"，无任何深度体现 |
| 1 | 技术理解有明显错误或硬伤 |

### 深度体现方式

- **性能优化**：具体的优化手段 + 前后对比数据 + 使用的分析工具
- **架构设计**：设计决策的权衡过程 + 解决了什么问题 + 有哪些取舍
- **源码贡献**：对开源项目的 PR、对框架内部实现的理解
- **疑难排查**：复杂 Bug 的定位过程 + 根因分析 + 修复方案
- **技术方案**：对比多种方案的优劣 + 最终选择的依据

## 6. 技术时效性

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 技术栈完全匹配 2025-2026 主流趋势，有前沿技术探索 |
| 4 | 主流技术为主，个别可更新 |
| 3 | 有一些过时技术混在其中 |
| 2 | 大量过时技术，主流技术覆盖不足 |
| 1 | 技术栈严重过时 |

### 2025-2026 前端主流技术参考

**框架**：React 18+、Vue 3、Next.js 14+、Nuxt 3
**语言**：TypeScript（已成为事实标准）
**构建**：Vite、Turbopack、Rspack
**状态管理**：Zustand、Pinia、Jotai、React Query / TanStack Query
**样式**：Tailwind CSS、CSS Modules、Styled Components
**测试**：Vitest、Playwright、Testing Library
**工程化**：pnpm、Turborepo、Monorepo、Changesets
**AI 相关**：Vercel AI SDK、LangChain.js（加分项）

**已过时或逐渐淘汰**：jQuery、AngularJS 1.x、Grunt/Gulp、Flow、Enzyme

## 7. 差异化亮点

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 有显著的差异化亮点（知名开源项目贡献/技术大会演讲/高星 GitHub 项目） |
| 4 | 有不错的亮点（技术博客/小型开源项目/团队内技术分享） |
| 3 | 有一些亮点但不够突出 |
| 2 | 几乎没有差异化内容 |
| 1 | 简历完全模板化，与其他候选人没有任何区分度 |

### 可作为亮点的内容

- 开源项目贡献（附 Star 数或下载量）
- 个人技术博客（附访问量或粉丝数）
- 技术大会演讲/社区分享
- 发表的技术文章（掘金/知乎/SegmentFault）
- 独立开发的产品（附用户量）
- 技术竞赛获奖
- 发布的 npm 包（附下载量）

## 8. 措辞与表达

### 评分标准

| 分数 | 标准 |
|------|------|
| 5 | 用词精准专业，动词有力，没有空话套话，信息密度高 |
| 4 | 整体表达良好，有少量可优化的措辞 |
| 3 | 有一些空话套话，或用词不够精准 |
| 2 | 大量空话套话，或频繁使用弱动词 |
| 1 | 表达不清、有语法错误、或过于口语化 |

### 动词替换指南

| 弱动词（避免） | 强动词（推荐） |
|---------------|---------------|
| 负责 | 主导、设计、构建、推动 |
| 参与 | 承担、落地、交付 |
| 做了 | 实现、优化、重构、搭建 |
| 完成了 | 交付、上线、推动落地 |
| 学习了 | 掌握、应用、实践 |

### 空话套话清单（应删除）

- "具有良好的团队合作精神和沟通能力"
- "学习能力强，能快速适应新环境"
- "热爱前端开发，对技术充满热情"
- "工作认真负责，有较强的责任心"

这些话每份简历都有，等于没说。用具体事例替代：比如用"推动团队采用 TypeScript，3 个月内全量迁移完成"来体现协作能力。

## STAR 法则项目改写指南

### 改写前（反面示例）

> **某电商平台**
> 技术栈：React + Redux + Ant Design
> 负责前端页面开发，完成了多个需求，包括商品列表、购物车、订单等模块。使用 React 和 Redux 进行状态管理。

### 改写后（正面示例）

> **某电商平台** | 前端核心开发 | 2024.03 - 2024.12
>
> 面向 C 端用户的跨境电商平台，月活 50 万+，前端团队 6 人。
>
> - **[S+T]** 商品列表页加载 2000+ SKU 时出现严重卡顿（FPS 降至 15 以下），用户投诉率上升 20%
> - **[A]** 采用虚拟列表（react-virtualized）+ 图片懒加载 + IntersectionObserver 方案重构列表渲染逻辑；引入 React Query 替代手写 Redux 异步逻辑，减少 60% 的模板代码
> - **[R]** 列表页 FPS 从 15 提升至稳定 58+，首屏加载时间从 3.2s 降至 1.4s；页面跳出率降低 18%
>
> - **[S+T]** 购物车模块需要支持跨页面实时同步、优惠券自动计算、库存校验
> - **[A]** 设计基于 Zustand 的购物车状态管理方案，结合 WebSocket 实现多标签页实时同步
> - **[R]** 购物车模块零重大 Bug 上线，支撑双 11 峰值 5 万+ 并发下单

## 不同年限的评审侧重

### 应届生 / 实习

- 教育背景放在靠前位置
- 项目不要求生产级，但要完整（从需求分析到上线部署）
- 重点看技术热情：有博客、有 GitHub 项目、参加过竞赛是强加分
- 基础是否扎实比技术栈是否丰富更重要
- 实习经历比课程项目更有价值

### 1-3 年

- 是否从"完成需求"进化到"独立承担模块"
- 技术选型是否有自己的思考（为什么选 A 不选 B）
- 是否有过性能优化或工程化改进的经历
- 项目描述是否能体现解决问题的能力

### 3-5 年

- 技术深度：性能优化、架构设计、复杂问题排查
- 团队影响力：制定规范、推动技术升级、代码审查
- 业务理解：是否理解技术决策对业务的影响
- 视野广度：跨端经验、全栈能力、新技术调研

### 5 年以上

- 技术领导力：架构方案的设计与推动落地
- 团队管理：带队规模、管理风格、人才培养
- 业务敏感度：技术选型与业务目标的对齐
- 行业视野：对前端趋势的判断和布局
